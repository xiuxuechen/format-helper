# format-helper

`format-helper` 是一套面向 Word/DOCX 的格式治理框架：它可以从一份“格式正确的标准 Word”中提炼规则，再用这套规则去审计、修复、复核另一份 Word，并生成可读的中文报告。

更直白地说，它不是一个“批量改字体字号”的小脚本，而是一套把 **标准文档 → 可确认规则 → 安全修复 → 复核验收 → 报告交付** 串起来的 AI 文档格式工程流程。

---

## 为什么需要这个框架

很多 Word 格式问题看起来很简单，实际处理时却很容易失控：

- 标题、正文、表格、目录、页眉页脚、分页规则分散在不同位置；
- “标准文档里看起来是这样”不等于规则已经被完整提取；
- AI 如果直接改 Word，容易出现不可追溯、不可复核、修坏原文档的问题；
- 修完以后如果没有二轮复核，很难确认到底修好了多少；
- 只输出 JSON/YAML 对普通用户不友好，真正需要的是能看懂的结论和交付物。

`format-helper` 的目标是把这些不确定性收敛成一条稳定流程：先建规、再确认、再修复、再验收；能自动处理的自动处理，不能确定的明确停下来让人确认。

## 适合谁使用

| 用户 | 典型诉求 |
| --- | --- |
| 文档处理人员 | 把待处理 Word 按某份标准 Word 的格式统一起来 |
| 审计、交付、项目管理人员 | 批量检查文档格式问题，并拿到可读报告 |
| 规则维护人员 | 从标准 Word 中沉淀可复用的格式规则包 |
| AI/Codex 工作流维护者 | 需要一个可审计、可恢复、有 Gate 的 DOCX 格式处理框架 |

如果你的需求只是“把这一段文字改成三号黑体”，直接手工改更快；如果你需要让一批文档稳定地遵循同一套格式标准，这个框架更合适。

---

## 核心理念

### 1. 标准文档是规则来源

你不需要一开始手写一整套格式规范。框架会先读取一份格式正确的标准 `.docx`，抽取标题、正文、列表、表格、目录、页面设置等格式事实，再生成可复用的规则包。

### 2. 人确认规则，机器执行规则

AI 可以帮助归纳规则，但不能把不确定内容悄悄当成确定规则。低置信度、冲突样式、缺失槽位和高风险动作都会进入 Gate，由人确认后再继续。

### 3. 不直接覆盖原始 Word

框架不会在原始 `.docx` 上直接动刀。输入文档会进入运行目录，修复动作只作用于工作副本，最终交付新的 Word 文件。

### 4. Markdown 给人看，JSON/YAML 给机器追溯

普通用户主要看 `RULE_SUMMARY.md`、最终 Word 和中文报告；机器可读的 JSON/YAML 用于审计、恢复、回归测试和问题定位。

### 5. 可恢复、可复核、可验收

每次运行都有 `run_id`，关键阶段会写入状态和日志。中断后可以恢复，修复后会做二轮复核，并生成最终验收报告。

## 快速开始

> 这个仓库的主要使用方式不是运行一个独立 CLI，而是在支持 `.codex/skills/` 的 AI 编码环境中调用 `format-helper` 技能。

### 1. 准备一份标准 Word

把格式正确的 `.docx` 放到当前工作区。第一次使用时，建议先只做建规，不要直接修复目标文档。

示例提示词：

```text
请使用 format-helper 为这份标准 Word 建规，生成规则摘要给我确认。
```

框架会生成规则包和人类可读的规则摘要：

```text
format-rules/{rule_id}/RULE_SUMMARY.md
```

你需要重点确认：

- 标题层级是否识别正确；
- 正文、列表、表格、目录等规则是否符合预期；
- 是否存在“未能确定”的格式属性；
- 是否有必须人工确认的冲突或风险项。

### 2. 确认规则

如果规则摘要没有问题，可以告诉 AI 继续使用这套规则。

示例提示词：

```text
规则摘要确认通过，请使用这套规则处理待修复 Word。
```

如果发现规则不完整，也可以直接补充人工确认：

```text
正文段落行距应为 1.5 倍，表格内容应为宋体 10.5pt、单倍行距。
```

### 3. 审计待处理 Word

如果你只想先看问题，不希望立即修复，可以执行审计模式。

```text
请使用 format-helper 按刚才确认的规则审计这份待处理 Word，先不要修复。
```

审计会告诉你哪些地方不符合规则，以及哪些问题可以自动修复、哪些需要人工确认。

### 4. 执行安全修复

确认可以修复后，再让框架生成修复计划并执行白名单动作。

```text
请使用 format-helper 按已确认规则修复这份待处理 Word，并生成最终报告。
```

最终常见交付物包括：

```text
format_runs/{run_id}/output/{原文件名}{yyyyMMddHHmm}.docx
format_runs/{run_id}/reports/FINAL_ACCEPTANCE_REPORT.md
```

### 5. 中断后恢复

如果流程中断或需要继续某次运行，可以用 `run_id` 恢复。

```text
请恢复 run_id=repair_20260513_xxx 的 format-helper 流程。
```

框架会读取运行状态，检查关键产物是否存在，并从最近的安全阶段继续。

## 典型工作流

```text
标准 Word
  ↓
事实快照：抽取文档中的客观格式事实
  ↓
语义归纳：识别标题、正文、表格、目录等角色
  ↓
规则打包：生成 format-rules/{rule_id}/
  ↓
规则确认 Gate：人确认规则是否可信、完整、可执行
  ↓
待处理 Word
  ↓
格式审计：找出不符合规则的位置
  ↓
修复计划：生成可校验、可追溯的动作清单
  ↓
修复确认 Gate：确认高风险或不确定动作
  ↓
安全写回：只对工作副本执行白名单修复
  ↓
二轮复核：检查修复是否生效、有无新增问题
  ↓
最终报告：交付 Word、报告和验收结论
```

## 运行模式

| 模式 | 适用场景 | 主要产物 |
| --- | --- | --- |
| `extract-rule` | 从标准 Word 建立规则 | `format-rules/{rule_id}/RULE_SUMMARY.md` |
| `audit-only` | 只检查待处理 Word，不修复 | 审计结果、问题清单 |
| `repair` | 审计后生成修复计划并安全写回 | 修复后 Word、复核结果、最终报告 |
| `resume` | 从已存在的 `run_id` 继续 | 更新后的运行状态和后续产物 |

## 你会拿到什么

### 面向用户的交付物

| 交付物 | 作用 |
| --- | --- |
| `RULE_SUMMARY.md` | 用中文解释规则包里到底包含哪些格式规则、哪些需要确认 |
| 修复后 `.docx` | 最终可交付的 Word 文件，不覆盖原文件 |
| `FINAL_ACCEPTANCE_REPORT.md` | 最终验收报告，说明修复结果、残留问题和人工确认项 |
| `MANUAL_CONFIRMATION.md` | 当存在未解析规则或风险动作时，给用户看的确认材料 |

### 面向追溯和恢复的内部产物

| 产物 | 作用 |
| --- | --- |
| `snapshots/*.json` | 记录 Word 的客观格式事实 |
| `semantic/*.json` | 记录语义角色、槽位事实和规则草案 |
| `plans/*.yaml` / `plans/*.json` | 记录执行计划、修复计划、人工确认项 |
| `logs/state.yaml` | 保存运行状态，用于恢复 |
| `logs/final_acceptance.json` | 保存最终验收的机器可读结论 |

普通用户通常只需要看第一组；第二组主要用于排查问题、恢复运行和做回归验证。

## `format-rules/` 和 `format_runs/` 的区别

这是使用本框架时最重要的边界之一。

| 目录 | 存什么 | 生命周期 |
| --- | --- | --- |
| `format-rules/{rule_id}/` | 可复用的规则包 | 可以被多次运行引用 |
| `format_runs/{run_id}/` | 某一次建规、审计、修复的运行产物 | 用于本次追溯、恢复和验收 |

规则包不应该写进某次运行目录里；运行目录只保存本次使用了哪个规则、做了哪些动作、产生了哪些结果。

## Gate 机制

`format-helper` 不追求“无脑跑完”，而是追求“该停的时候停”。

| Gate | 什么时候触发 | 为什么重要 |
| --- | --- | --- |
| 规则确认 Gate | 建规后 | 防止缺失槽位、冲突样式或低置信度规则被直接用于修复 |
| 规则选择 Gate | 修复前 | 确认待处理 Word 使用的是哪套规则 |
| 参数覆盖 Gate | 用户临时指定规则参数时 | 防止临时修改和规则包事实混淆 |
| 修复确认 Gate | 执行修复前 | 高风险或不确定动作必须先确认 |
| 验收 Gate | 修复后 | 确认输出文件、复核结果和报告都满足交付条件 |

这套机制会让流程看起来比普通脚本“慢半拍”，但它换来的是可解释、可追溯和可恢复。

---

## 安全边界

框架默认遵守以下约束：

- 不覆盖原始 `.docx`；
- 不删除用户文件或历史运行目录；
- 不跨工作区写入；
- 不把高风险、低置信度、未确认动作直接执行；
- 不把内部临时 Word 当作最终交付物；
- 最终 Word 统一输出到 `format_runs/{run_id}/output/`；
- 如果当前环境无法刷新 Word 自动目录，流程应阻塞或提示人工处理，而不是把占位目录当作已验收结果。

## 与普通脚本的区别

| 普通格式脚本 | format-helper |
| --- | --- |
| 直接按固定参数改文档 | 先从标准文档抽取并确认规则 |
| 通常只处理字体、字号等少量字段 | 覆盖标题、正文、列表、表格、目录、页面等多类元素 |
| 改完就结束 | 修复后还有二轮复核和最终验收 |
| 失败后难恢复 | 通过 `run_id`、状态文件和产物目录恢复 |
| 输出偏机器日志 | 输出人类可读规则摘要和中文报告 |
| 容易覆盖源文件 | 默认只处理副本，不覆盖原始 Word |

---

## 最佳实践

1. **先建规，再修复**：第一次处理某类文档时，不要跳过规则摘要确认。
2. **优先使用高质量标准文档**：标准 Word 本身越规范，抽取出的规则越稳定。
3. **认真看未确认项**：未解析槽位、冲突样式和高风险动作是最容易影响最终质量的地方。
4. **保留 `run_id`**：出现问题时，`run_id` 是恢复、排查和复核的入口。
5. **不要把内部产物当交付物**：最终交付以 `output/` 下规范命名的 Word 和 `reports/` 下中文报告为准。
6. **规则包要复用，不要复制运行目录**：长期资产放在 `format-rules/`，单次过程放在 `format_runs/`。

---

## 一句话总结

`format-helper` 试图解决的不是“怎么让 AI 改 Word”，而是“怎么让 AI 在可确认、可追溯、可恢复、可验收的边界内治理 Word 格式”。
