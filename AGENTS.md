本文件是仓库级 Codex 指令入口。

## 目标

使用 Codex-only 的 `.codex/skills/` 路径加载本框架的全部能力，不依赖根目录历史 `skills/` 结构。

## 能力承载

- 自动触发能力：`.codex/skills/`


## 代码规范约束

如果项目中存在 `docs/_foundation/_planning/06_CODE_STANDARDS.md`，则在编码、重构、代码评审、测试补齐和修复建议时，应优先将其作为项目级代码规范单一事实源。

执行原则：
- 优先遵循该文档，再结合现有代码风格落地
- feature 文档可以增加更严格规则，但不应与其冲突
- 若现有代码与该规范不一致，应先说明差异，再决定沿用旧模式或推动规范升级
- 涉及 Git 提交时，提交前先检查其中的提交约定，默认使用中文提交信息

## 执行权限策略

以下操作默认授权，无需额外确认：

| 操作 | 限制 |
|------|------|
| 读取工作区内 `.docx` | 仅限用户提供、fixture、运行目录或规则目录 |
| 创建 `format-rules/` | 幂等创建 |
| 创建 `format_runs/{run_id}/` 子目录 | 幂等创建，见下方目录预创建流程 |
| 写 snapshot、语义 JSON、计划、日志、报告 | 仅写运行目录 |
| 写内部 skill 结果信封和摘要 | 仅写 `logs/skill_results/` |
| 写新规则包 | 仅写 `format-rules/{rule_id}/` |
| 写修复后 Word 副本 | 不覆盖原始 Word |
| 刷新自动目录字段 | 仅在运行目录输出副本上执行 |
| 生成规范命名交付物 | 仅在 `output/` 下生成 `{原文件名}{yyyyMMddHHmm}(_r[0-9]{2})?.docx` |

## 仍需人工确认的动作

以下操作必须人工确认：

- 删除文件或清理历史目录
- 覆盖原始 `.docx`
- 覆盖已有 active 规则包
- 写入工作区外路径
- 执行未在技能文档或脚本白名单中的命令
- 执行高风险、低置信度或人工确认未完成的修复动作

## 工作流输出合规

如果当前请求触发了 `.codex/*.md`、`SKILL.md`、playbook 或其他仓库内工作流文档，并且其中定义了任何面向用户可见的要求，包括但不限于：执行步骤、固定话术、欢迎语、输出顺序、建议输出结构、固定分块、示例输出或其他等效模板，则这些内容一律视为强约束，必须体现在最终输出中，而不只是参考。

凡执行步骤中要求“显示”“说明”“列出”“输出”“提示”给用户的内容，均属于最终答案的必需组成部分，不能仅在内部执行而不对用户展示。

除非工作流文档明确标注某项为“可选”，否则不得：
- 省略
- 调整顺序
- 改写成自由总结
- 用更简短的话替代固定话术
- 只完成内部动作而不按要求输出给用户

执行这类工作流时，最终回答前必须逐项自检：
1. 是否已覆盖所有面向用户可见的执行步骤
2. 是否保留要求的固定分块或标题
3. 是否遵守指定顺序
4. 缺失内容是否已明确标注为“无”或说明原因

若以上任一条件未满足，则视为该工作流未完成，不能提交最终答案。

## 编码安全

对文件进行读取、编辑或写回前，必须先确认文件编码格式；修改后的文件必须与原有内容编码保持一致，不得擅自改变编码。

- 未确认编码前，不得整文件重写或覆盖原文件。
- 先判断当前运行环境是 Windows、Linux 还是 macOS，再选择该环境下最稳定的读取和校验命令。
- 在 Windows 环境下，优先使用 `rg` 做内容核对，使用显式指定编码的原始读取方式确认真实内容。
- 在 Linux 或 macOS 环境下，优先使用该平台原生且稳定的工具链进行编码和内容检查，不沿用其他平台的命令习惯。
- 必须先区分“终端显示乱码”和“文件内容编码错误”是否为同一问题；只有确认原始内容异常后，才能执行重写、转码或整文件修复。
